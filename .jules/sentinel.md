## 2024-05-24 - Secure File Creation Pattern (TOCTOU Prevention)
**Vulnerability:** A Time-of-Check to Time-of-Use (TOCTOU) vulnerability existed in `src/logger.py` where a log file was created using `Path.touch(exist_ok=True)` and subsequently its permissions were changed using `os.chmod(self.log_file, 0o600)`.
**Learning:** This approach leaves a brief window between the file's creation and the permissions change where a malicious actor could access or modify the file.
**Prevention:** Instead of separating the creation and permission modification, use `os.open` with the `os.O_CREAT` flag and directly specify the desired permissions (e.g., `0o600`). This ensures the file is created atomically with the correct permissions. For example: `fd = os.open(file_path, os.O_CREAT | os.O_APPEND | os.O_WRONLY, 0o600)` followed by `os.close(fd)`.

## 2024-05-24 - Input Validation Bounds Checking Bypass (NaN / Inf)
**Vulnerability:** The input validation logic in `src/validation.py` for variables like quantity, price, stop price, and percentage was vulnerable to bypasses because it converted inputs to `float` but did not explicitly reject special float values like `float('nan')` or `float('inf')`. Because `nan <= 0` and `nan > MAX` always evaluate to `False`, these values could bypass bounds checks and propagate into business logic or API calls.
**Learning:** Python's floating-point handling rules for special values like `NaN` mean that standard relational operators (`<`, `<=`, `>`, `>=`) behave non-intuitively (evaluating to `False` for `NaN`). This can silently bypass bounds checking and validation loops if input is not explicitly validated as a standard number.
**Prevention:** Always use `math.isfinite(val)` after converting an input to a `float` to explicitly reject `NaN` and `Infinity` before performing bounds checks (`val > 0`, `val < MAX`) and progressing to subsequent business logic.
